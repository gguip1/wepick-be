# 2. 코딩 컨벤션 및 주요 패턴

이 문서는 프로젝트의 코드 일관성을 유지하고, 모든 AI 에이전트가 동일한 스타일과 패턴을 따르도록 하기 위한 규칙을 정의합니다. **모든 에이전트는 코드를 작성하거나 수정하기 전에 반드시 이 문서를 숙지해야 합니다.**

## 2.1. 핵심 아키텍처 패턴 (Mandatory Patterns)

### 2.1.1. 인증: 커스텀 세션 기반 인증

이 프로젝트는 Spring Security의 표준 인증 메커니즘을 사용하지 않습니다. 대신 다음과 같은 독자적인 흐름을 따릅니다.

1.  **`SessionAuthFilter`**: HTTP 요청의 세션에서 `userId`를 추출합니다.
2.  **`SecurityContext`**: 추출된 `userId`를 `ThreadLocal`에 저장하여 요청 처리 동안 유저 정보를 전역적으로 사용할 수 있게 합니다.
3.  **`@Auth` 어노테이션**: 인증이 필요한 컨트롤러 메소드에 이 어노테이션을 붙입니다.
4.  **`AuthInterceptor`**: `@Auth`가 붙은 메소드가 호출되기 전에 `SecurityContext`에 `userId`가 있는지 검사하여 인증 여부를 확인합니다.

**규칙**:
- 새로운 API에 인증이 필요할 경우, Spring Security 설정을 변경하지 말고 컨트롤러 메소드에 `@Auth` 어노테이션을 추가해야 합니다.
- 인증된 사용자의 정보가 필요할 경우, `SecurityContext.getUserId()`를 호출하여 `userId`를 가져와야 합니다.

### 2.1.2. 예외 처리: `GlobalExceptionHandler`

- 모든 비즈니스 로직 상의 예외는 `GlobalExceptionHandler`에서 처리됩니다.
- 예상된 예외 상황(예: "게시글을 찾을 수 없음")이 발생하면, `ErrorCode` Enum에 정의된 값을 사용하여 `ErrorException`을 발생시켜야 합니다.

**규칙**:
- `try-catch` 블록으로 예외를 직접 처리하여 임의의 에러 메시지를 반환하지 마십시오.
- 새로운 예외 유형이 필요하면, `ErrorCode`에 적절한 상태 코드와 메시지를 정의하고 `ErrorException`을 사용하십시오.

## 2.2. 불일치 문제 및 해결 방안 (Inconsistencies & Solutions)

코드베이스 분석 결과, 다음과 같은 불일치 문제가 발견되었습니다. 코드를 수정할 때는 다음 가이드라인에 따라 일관성을 개선해야 합니다.

### 문제: `User` 엔티티의 Soft Delete 로직 불일치

- **현황**: 사용자가 탈퇴할 때(`softDelete` 메소드), 재가입을 허용하기 위해 이메일(`email`) 필드의 값을 변경하는 로직이 존재합니다. 하지만, 동일하게 고유 값(unique constraint)이어야 하는 닉네임(`nickname`) 필드에 대한 처리 로직은 주석 처리되어 있습니다.
- **원인**: 기능의 초기 개발 단계에서 닉네임 정책이 확정되지 않았거나, 구현이 완료되지 않은 것으로 추정됩니다.
- **위험**: 이 상태에서는 탈퇴한 사용자와 동일한 닉네임으로 다른 사용자가 가입할 수 없어 UX 문제를 야기하거나, 상황에 따라 데이터베이스 제약 조건 위반 오류를 발생시킬 수 있습니다.

**해결 규칙**:
- **(필수)** `User` 엔티티의 `softDelete` 관련 기능을 수정할 때는, 주석 처리된 닉네임 변경 로직을 활성화하고 이메일 처리 방식과 동일하게 일관된 방식으로 구현해야 합니다. 예를 들어, 닉네임 뒤에 타임스탬프나 랜덤 문자열을 추가하는 방식을 적용합니다.
- **(권장)** 해당 로직 수정 시, 관련 테스트 케이스(`User` 관련 테스트)를 반드시 추가하여 soft delete 이후 이메일과 닉네임이 정상적으로 변경되는지, 동일한 정보로 재가입이 가능한지 검증해야 합니다.
